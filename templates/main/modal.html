Remote Heating Control

Around the time i started this project, hive heating systems were first being advertised (for quite a price may i add) so i wondered if the same functionality could be achieved for a lot less.

The site is hosted on one of my Raspberry Pi 3's located above my boiler. When a functional button is pressed a signal is sent asynchronously (meaning in the background, without a page refresh) to the server, which then does what is required to fulfil the task. Sometimes this is as simple as toggling a GPIO pin which is connected to a relay, thus turning whatever the relay is wired to on (like the main heating button). Other times it could be more complicated than this, such as sending a command to another raspberry Pi located in another room (or even another house) which then in turn carries out its own procedure to complete the task.

In its current state, my home control system allows control of:
-My heating (along with temperature control through an Arduino which controls a stepper motor connected to a heating knob)
-My mother's heating (located elsewhere so my Pi has to send commands over an SSH connection)
-The lights in my room (also connected to another Pi so commands have to be sent through SSH again)
-A set of addressable LEDs around my monitors in my room (can be turned on, off and have the colour changed also. The colour is changed through sending commands across a USB serial connection)
-My desktop PC can be turned on or off (through a relay connected the header of its motherboard which toggles for 1 second, recreating a button press)
-Some spare PC fans i had after building my PC which are now suspended above my monitors to keep me cool in the summer

Othello.js

Board games are great entertainment but the relatively recent rise of technology has made them obsolete for many applications; They're not portable enough to be played anywhere except at home and loosing pieces easily ruins the game. A digital version would suffer from none of these issues while still offering the same levels of entertainment. This is why I decided to make a web-based version of Othello (also known as Reversi).

The main objective of Othello is to own as many pieces on the board as possible at the end of the game. This can be achieved by surrounding an opponent's piece (or pieces) with one of yours either side (this works vertically, horizontally and diagonally).

My interpretation of Othello is produced in pure JavaScript. It works by creating the html board by iterating over its rows and columns. Then, it assesses the board and allows all squares that would be an acceptable move clickable and changes the squares colour appropriately to illustrate this to the user.

Text Based Python Game

To grasp Object Oriented Programming concepts more thoroughly, we decided to make a short text based game in python; after all the best way to learn in programming is by doing.

In all honesty, the hardest part about making the game was coming up with names for characters and what they should say, particularly when doing it on something with such a massive fanbase of devoted enthusiasts like the lord of the rings.

As it was a text based game, we had to devise a less conventional movement and interaction system compared to modern 3D visual games that we see commonly today.

Another problem we faced was difficulty, the player is not required to skillfully maneuver a car around a racetrack or shoot a baddie in the face from afar as its a text game; so how do we stop it from becoming boring due to a lack of challenge?

Our solution was to not reveal excessive information, to keep the player thinking, and to make it perma-death with one wrong move instantly causing the end for your character and a fresh start required, so that there is still a buzz about getting through a section.

Virtual Christmas Cards

Christmas cards can be quite laborious and none of them ever seem particularly original. So, to solve such a monumental issue within society I decided to make a virtual christmas card hosted on my portfolio site. This card can be easily addressed with minimal effort and no code modifications.

It takes the value after ‘/christmas/’ in the url and replaces any underscores with a space and uses that value as the card’s recipient name. It also runs the name through a prefix algorithm i created to not treat the prefix as a first name so that the card is addressed to “Mr name,” not just “Mr,”.

The card itself is fairly simple, just a couple of rectangles that rotate on click with a shadow. Depending on the screen size, the card fills the screen differently so that it can be viewed on any form factor.

This website itself

After successfully implementing my home control website, I wanted to further my web development knowledge and as the best way to learn (when it comes to programming) is by doing, I decided to make this site.

The site was built using JavaScript (and jQuery) to handle the client-side actions (such as the animated text that cascades down on page load) and (a lot of) CSS to style the page. Bootstrap was also used to cleanly transition the page onto different screen form factors. AOS (Animate On Scroll) is a JavaScript library I used for all the scroll animations.

As you may have noticed, fairly bold colour scheme of red/orange and purple (my two favourite colours) is used but as I intended this site to appear professional, I made sure to refine the theme to still contain my ideal colours while not causing the user any visual discomfort.

I contemplated using another backend framework to my home control site (which currently uses Bottle, a python micro-framework), such as Django or Node.js, so that I would get a broader understanding of how everything works. However, as one of the intended uses for this site was to host demos of my other work and that would only be feasible with Bottle. Besides, I would rather gain a deeper knowledge of one framework so that I know it inside out and can solve more advanced problems before learning others.

It is hosted on my ubuntu server at home (so hopefully it doesn’t get extortionate levels of traffic else I might be getting phone calls from my dad complaining that he can't stream his shows) which runs Nginx (webserver software) to fetch the static files, such as images, much faster than the bottle program is able to. I also use Nginx to direct traffic to a few of my RaspberryPi’s which host other sites but due to the router being on a different TCP/IP layer to check domain names all port 80 traffic has to be sent to one machine which then directs unwanted traffic to its destination.

To reduce bandwidth consumption and, more importantly, increase page load speed, gzip is used to compress the page data before it is decompressed on the client side by the browser. Additionally, the images are all compressed prior to sending to further reduce load times and reduce the server’s workload. Overall, this caused a dramatic improvement in page load times (from about 10 seconds to 3 in my testing) and helped to improve my page’s search engine status, allowing it to be indexed above the many ‘Reiss’ (the clothing brand) links that appear when my name is searched.
